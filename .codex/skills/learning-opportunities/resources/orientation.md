# Repo Orientation: pi-bio-agent

> Generated by orient. Re-run to update.

## One-line purpose
`pi-bio-agent` is a TypeScript library and Pi extension for running scientific-agent work as manifest-declared resources, DuckDB SQL, content-addressed artifacts, and temporal provenance instead of per-question skills.

## Primary language(s)
TypeScript is dominant. Markdown documents the design and examples. R and shell appear in process-compute examples.

## Pipeline / workflow stages
1. A host loads a manifest or operation spec and binds explicit effect ports such as DuckDB, fetch, CAS, process runner, or store.
2. The registry validates declared resolvers, resources, operations, term sets, and strict manifest shape.
3. Resolvers materialize declared resources into DuckDB tables or handles, producing receipts and source snapshots.
4. Operations run one read-only SQL statement, optionally after schema discovery, scale materialization, and host bindings.
5. The host persists run records, result rows, receipts, replay specs, CAS artifacts, and optional `bio_observations` facts.
6. Memory, jobs, approvals, graph edges, and run facts are queried later as temporal data, not prompt-only context.

## Core concepts
**Manifest as program**: Manifests declare resources, resolvers, operations, and term sets; TypeScript interprets those declarations rather than encoding per-question biomedical skills. Lives in `src/core/manifest.ts`, `src/core/operations.ts`, and examples under `examples/`.

**DuckDB substrate**: Files, network responses, graph closure, and analysis reductions become tables and SQL. DuckDB extensions such as `duckhts` and `ducknng` are part of the product surface. Lives under `src/duckdb/`, `docs/duckdb-substrate.md`, and `docs/design.md`.

**Receipts, replay, and CAS**: Runs are not just answers; they carry content digests, resolver receipts, replay specs, and optional CAS-backed artifacts. Lives in `src/hosts/run-store.ts`, `src/core/cas.ts`, `src/hosts/action-cache.ts`, and related tests.

**Unified temporal store**: Memory notes, facts, jobs, approvals, graph edges, and run observations are rows in `bio_observations`, read as-of time and projected into graph views. Lives in `src/duckdb/observations.ts`, `src/hosts/memory-store.ts`, `src/hosts/job-store.ts`, and `docs/memory-and-knowledge-unification.md`.

**Host-owned effects**: The library records and validates effects, but the host grants network, process compute, DuckDB extension loading, CAS, and policy through dependency-injected ports. Lives in `src/core/ports.ts`, `src/hosts/run-store.ts`, and the Pi extension entrypoints under `extensions/pi-coding-agent/`.

## Common gotchas
- Do not edit `README.md` directly. Change `README.Rmd`, run `npm run readme:rmd`, then run `npm run readme:tools`.
- Read-only SQL validation is not a sandbox. `validateReadOnlySelect` is statement-class safety; egress and filesystem policy belong to the host.
- `duckdb.sql_materialize` is intentionally broad but still read-only. It is the general materialization primitive, not a place to hide writes or bespoke biomedical helpers.
